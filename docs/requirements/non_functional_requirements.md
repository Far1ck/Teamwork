<h1>Нефункциональные требования</h1>

<h2>NFR-01. Производительность</h2>
<p>
  <li>Время ответа для GET/recommendation/&ltuser_id&gt:
    <ul type = "circle">
      <li>&#8804 1 сек. для кешированных данных.</li>
      <li>&#8804 3 сек. для первых запросов (без кеша).</li>
    </ul>
  </li>
  <li>Кеширование результатов SQL-запросов (Caffeine).</li>
</p>

<h2>NFR-02. Надежность</h2>
<p>
  <li>Система доступна 99,9 % времени.</li>
  <li>Данные в H2 не изменяются (только чтение).</li>
  <li>PostegreSQL используется для записи с транзакциями.</li>
</p>

<h2>NFR-03. Безопасность</h2>
<p>
  <li>Доступ к API аутентифицируется (JWT/OAuth2 - в будущих терациях).</li>
  <li>SQL-запросы защищены от инъекций (использование параметризованных запросов).</li>
</p>

<h2>NFR-04. Масштабируемость</h2>
<p>
  <li>Архитектура позволяет добавлять новые типы правил бед передеплоя.</li>
  <li>Кеш масштабируется горизонтально (при необходимости).</li>
</p>

<h2>NFR-05. Логирование</h2>
<p>
  <li>Все запросы к API логируются (уровень INFO).</li>
  <li>Ошибки логируются с трассировкой (уровень ERROR).</li>
</p>

<h2>NFR-06. Мониторинг</h2>
<p>
  <li>Endpoint /managment/info возвращает версию сервиса.</li>
  <li>Статистика по правилам доступна через /rule/stats.</li>
</p>
